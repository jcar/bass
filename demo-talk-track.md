## Section 1 — Hook (~1 min)

**What to say:**
- "I'm going to show you a side project. Not because you're going to go fishing this weekend — because building it changed how I think about what AI is actually *for* in production systems."
- "30-second context: I fish. The hardest part of bass fishing isn't casting, it's deciding *what to throw* based on conditions. Pros have opinions. Conditions change every hour. It's a high-dimensional decision problem with a lot of folk knowledge."
- "So I built a thing. Let me show you."

**Speaker bullets:**
- Keep it short. Do not explain the architecture yet.
- Don't apologize for it being a side project. Own it.
- The seed of intrigue you're planting: "changed how I think about AI in production."

## Section 2 — Live demo (~5 min)

You are driving. They are watching. Narrate what you click, not what it does.

### Beat 1 — First impression (~45s)

- Open the site. Let it auto-detect / pick a lake.
- "This is Strikezone. It tells me where to fish, when to fish, and what to throw."
- Point at the top: **Confidence gauge / bite forecast** — "AI-tuned bite window."
- Point at the forecast strip — "Six days out. NOAA weather, USGS water data, solunar."

### Beat 2 — The "what to throw" card (~90s)

- Scroll to the **Top Pick / What To Throw** section.
- "This is the money shot. Given the conditions right now — water temp, clarity, pressure trend, frontal system — here's the primary lure, the alternate, the color, the retrieve, and *which pro would throw it.*"
- Click into the pro attribution. "KVD. Hackney. Wheeler. Every recommendation is tied to a specific human being who said this thing on the record."

### Beat 3 — Change a condition, watch it react (~60s)

- This is the wow moment. Pick something visible.
- Change **water clarity** from clear → muddy. "Watch the recommendations."
- They flip instantly. Different lure, different color, different pro quote.
- Flip through the **forecast days**. The frontal system shifts (stable → pre-frontal → post-frontal) and the whole briefing rewrites.
- "Notice how fast that is. Hold that thought."

### Beat 4 — Anglers page (~60s)

- Navigate to `/anglers` (or click an angler card).
- "Seven pros. Each one has a personality — their conditions, their go-to lures, their quotes."
- "You can 'follow' an angler and the whole app re-weights toward their philosophy."

### Beat 5 — The reveal (~45s)

- Switch to **DevTools → Network tab**.
- "Before I explain how, look at what's NOT happening here."
- Interact with the app — change conditions, flip days.
- "Zero calls to any LLM. No OpenAI, no Anthropic, no vector DB, no server. This is a static site on GitHub Pages."
- **Pause.** Let that land.

## Section 3 — The twist (~1 min)

**What to say:**
- "Every recommendation you just saw — the attribution, the pro quotes, the conditional logic — was *generated by an LLM.* Just not at the time you asked for it. Months ago, during a build step."
- "The mental flip I want to plant: **treat the LLM like a compiler, not an interpreter.**"
- "Compiler = expensive once, cheap forever. Interpreter = expensive every single request. Most of us are defaulting to interpreter because that's what the RAG tutorials showed us."

**Speaker bullets:**
- Say "compiler, not interpreter" out loud. It's the line you want them repeating on Monday.
- Resist the urge to caveat ("obviously RAG is still good when…"). Save that for Q&A.

## Section 4 — The pipeline (~4 min)

Pull up the repo. `knowledge/scripts/` directory listing.

### The four layers (whiteboard or screen-share)

Draw or show:

```
[472 articles from 7 pro anglers]
         │   (Python scraper — requests + BeautifulSoup)
         ▼
[structured extraction]            ← LLM call #1 (per article)
         │   Claude with a strict JSON schema:
         │   { lure, conditions, tipRules[], proSource }
         ▼
[1,213 knowledge entries + 108 lure opinions]
         │
         ▼
[condition enrichment]             ← LLM call #2 (per entry)
         │   Second pass adds *implied* conditions the first
         │   pass missed (frontal pressure, depth, structure)
         ▼
[1,213 enriched, condition-tagged entries]
         │   Python — pure scoring + retrieval, zero LLM
         ▼
[326 pre-generated tactical briefings]
         │   6 seasons × 3 clarities × 3 fronts × 7 lure
         │   categories, minus impossible combos
         ▼
[~1MB JSON bundle, imported at build time]
         │
         ▼
[Next.js static export → GitHub Pages]
         │   Runtime lookup: getBriefing(season, clarity,
         │   front, lure). Map lookup. O(1). No network.
         ▼
[You, on a fishing dock, with no cell signal]
```

### Speaker bullets

- **Layer 1 — scrape.** "Dumb Python. 472 articles from bassmaster.com, wired2fish, majorleaguefishing. No AI involved. The old-fashioned part of the stack."

- **Layer 2 — extract (job 1 of 2).** "Here's where the LLM earns its paycheck. I don't ask 'summarize this article.' I give it a strict schema and say: *fill it in.* Lure, conditions, tipRules array, priority, pro attribution. The LLM becomes a natural-language-to-JSON compiler. No summarization — the JSON *is* the output."

- **Layer 3 — enrich (job 2 of 2).** "Two LLM passes, different jobs. First extraction only tags what's *explicitly stated*. An article about 'cranking offshore ledges in August' doesn't say the word 'summer' or 'deep' or 'structure:ledge' — but a human reading it knows. A second LLM pass reads each entry with full comprehension and fills in the implied conditions. This is the layer that replaced a regex keyword-match pass I started with — regex kept firing false positives on substrings like 'switch' matching 'twitch.'"

- **Layer 4 — generate.** "Now it's pure Python. For every valid condition combination, score and rank the matching knowledge entries, cap per angler for diversity, assemble a briefing. 326 JSON files. Bundle them. Ship."

- **Runtime.** "The app imports the bundle at build time. `getBriefing(season, clarity, front, lure)` is a dictionary lookup. That's why clarity flipped instantly when I changed it — no retrieval, no synthesis, no token budget."

### Numbers to have ready

| Stage | Input | Output | LLM? |
|---|---|---|:-:|
| Scrape | 472 URLs | 472 text files | ✗ |
| Extract | 472 articles | 1,213 entries + 108 opinions | ✓ once |
| Enrich | 1,213 entries | 1,213 enriched entries | ✓ once |
| Generate | 1,213 entries | 326 briefings (~1MB) | ✗ |
| **Runtime** | **user query** | **instant briefing** | **✗** |

## Section 5 — Why this matters / where it generalizes (~3 min)

**What to say:**
- "The pattern isn't 'bass fishing.' The pattern is: **corpus of expert knowledge + bounded condition space = pre-compile the intersections.**"
- "RAG is the right call when the question space is unbounded. Any question, any document. But a huge number of the use cases we actually ship at work aren't unbounded — they're filterable. If you can enumerate the axes your users will slice on, you can compile."

**Seed ideas for Valent applications** *(pick 1–2 that are most relevant to your firm's actual work — I don't know exactly what Valent does, so swap these):*

- **Investment memos / deal patterns.** Sector × stage × deal type × thesis. Compile precedent answers from our internal corpus.
- **Diligence checklists.** Asset class × jurisdiction × risk flag. Expert knowledge from past deals, pre-synthesized.
- **Operating playbooks.** Portfolio company × function × growth stage. "What did we do the last time a portco hit this inflection?"
- **Internal support / runbooks.** Product × environment × error class. Ship the compiled playbook to the responder instantly.

**The test I use now before reaching for RAG:**
1. Can I enumerate the dimensions? (If yes → consider compile-time.)
2. Is the corpus updated hourly, or quarterly? (Quarterly → compile.)
3. Do I need attribution? (Attribution is *built into* a schema-first pipeline. It's a pain to retrofit into RAG.)
4. Do I need sub-second latency? (Static lookup wins every time.)

**Speaker bullets:**
- The RAG-vs-compile contrast isn't "RAG bad." It's "picking the wrong tool is a large quality and cost penalty."
- Attribution is the sleeper win — every briefing knows which pro said which thing because the schema forced it. That's hard to get from retrieval.
- Cost story: two one-time batch runs against Claude vs. a per-query token bill for the lifetime of the product. For bounded domains, the math isn't close.

## Section 6 — Close (~1 min)

**What to say:**
- "Three things to take away:"
  1. "**LLM is a build tool, not a runtime dependency** — for any domain you can enumerate."
  2. "**Schema-first extraction** beats free-form summarization. The JSON is the product; the prose is the raw material."
  3. "**If your users filter on N dimensions, you can precompile the cross-product.** It's almost always cheaper than you think."
- "I wrote the architecture up in more detail — link's in Slack. Happy to pair with anyone who wants to try this on a real Valent workflow."
- "That's the show. Questions?"

## Likely Q&A — have answers ready

**Q: How do you update the data?**
A: Re-run the pipeline. It's idempotent and takes ~an hour for the full corpus. New angler = one runbook, drop-in. I have a markdown runbook (`ANGLER-EXTRACTION-RUNBOOK.md`) that another Claude Code session can execute autonomously.

**Q: What about long-tail queries that don't fit your 326 combos?**
A: They don't exist in this domain — the axes are complete. For a domain where they do exist, you ship the static bundle *and* keep a RAG fallback for the long tail. Hybrid works.

**Q: Why not just embed the articles and do RAG?**
A: Tried it mentally. Three problems: (1) chunks split mid-thought, lose condition context; (2) attribution is a hack; (3) every query burns tokens forever. Compile-time solved all three.

**Q: How much did this cost in API spend?**
A: Extraction + enrichment were batch jobs against Claude. Low double-digit dollars total. Zero marginal cost per user now.

**Q: What model did you use and why?**
A: Claude (Opus for extraction because schema adherence matters; Claude Code for the enrichment pass because I wanted file-level iteration without babysitting an API loop).

**Q: How do you handle conflicting pro opinions?**
A: Preserved, not averaged. Each briefing surfaces multiple pros by name. The user picks the voice they trust. Conflict *is* the signal — muting it would destroy the value.

**Q: Is this a product?**
A: No. It's a pattern. The product happens to be fishing.
